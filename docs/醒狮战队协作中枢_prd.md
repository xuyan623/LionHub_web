## 2. 产品背景

醒狮战队直属于机器人协会，是专为 RoboMaster 赛事设立的技术团队。战队由管理层、技术组、运营组三部分构成，其中技术组包含机械组、电控组、算法组，并围绕步兵、哨兵、英雄、工程、无人机、飞镖、雷达等兵种机器人组建跨方向协作团队。

当前战队成员类型包括正式队员、梯队队员、育苗队员。成员在战队中的贡献既包含技术学习、研发产出，也包含场地建设、运维维护、宣传运营、比赛筹备等事务性劳动。因此，系统需要同时支持成员管理、任务公开领取、任务委派、跨组协作、贡献量化、晋升考核和团队绩效分析。

本产品旨在将战队任务从零散沟通转为结构化管理，并通过"研习点 / 工时点 / 管理点"体系记录成员成长与贡献。

---

## 3. 产品定位

**醒狮战队协作中枢**是面向 RoboMaster 战队的内部协作平台，用于统一管理成员、任务、任务市场、积分结算、兵种项目进度与绩效统计。

它不是通用 Todo 工具，而是围绕战队真实协作方式设计的专用系统，重点解决：

1. 成员信息分散，身份、组别、技能、晋升状态不清晰。
2. 任务发布、领取、退出、审核缺乏标准流程。
3. 任务贡献无法公平量化，尤其是技术学习与事务劳动难以同时记录。
4. 成员负载不可见，任务容易集中到少数人身上。
5. 兵种项目进度缺少统一总览。
6. 育苗、梯队、正式队员的成长路径缺少数据沉淀。

## 4. 产品目标

### 4.1 核心目标

1. 建立战队成员数据库，记录成员身份、部门、方向、兵种组、技能标签和贡献数据。
2. 建立完全公开的任务市场，让所有成员都可以查看任务并主动参与。
3. 支持正式队员发布任务，管理员负责最终审核和点数结算。
4. 支持任务委派、领取、退出、中途加入、组长审批等战队协作场景。
5. 通过研习点、工时点、管理点记录成员贡献。
6. 提供成员负载、任务进度、部门贡献、兵种项目进度和团队绩效概览。
7. 为育苗队员、梯队队员、正式队员的晋升和考核提供数据依据。

### 4.2 非目标

V1 阶段暂不重点实现以下内容：

1. 不做复杂即时通讯系统。
2. 不替代代码仓库、图纸管理系统或云盘。
3. 不做财务报销全流程，仅记录相关任务与基础附件。
4. 不做复杂 OKR / KPI 系统。
5. 不做多社团 SaaS 化管理，优先服务醒狮战队单组织场景。

## 5. 用户角色与权限

### 5.1 成员身份

| 身份 | 说明 |
|---|---|
| 指导老师 | 代表学院或学校监督战队重大事项，主要查看整体数据和项目进展 |
| 管理层 | 包括队长、副队长、项目管理、质量管理等，负责战队整体管理和任务审核 |
| 组长 / 方向负责人 | 包括机械组、电控组、算法组、运营组及细分方向负责人，负责本方向任务审批与成员指导 |
| 正式队员 | 可发布任务、参与任务、带新、负责具体技术或运营事务 |
| 梯队队员 | 可领取任务、参与任务、协助正式队员完成工作 |
| 育苗队员 | 可领取所有公开任务，但高难度技术任务需要审批后加入 |
| 游客 / 待审核用户 | 注册后未通过审核的用户，只能等待审核或完善资料 |

### 5.2 系统权限角色

| 权限角色 | 权限范围 |
|---|---|
| 管理员 | 管理成员、审核任务、结算点数、调整分配比例、处理补偿点 |
| 组长 | 审批所属方向的高难任务加入申请，查看本组成员与任务数据 |
| 正式队员 | 发布任务、领取任务、更新自己参与的任务 |
| 普通成员 | 查看所有任务、领取公开任务、更新自己参与的任务 |
| 指导老师 | 查看整体统计、成员概况、兵种进度，不参与普通任务编辑 |

### 5.3 权限原则

1. 所有人都可以查看任何其他人的任务。
2. 管理员可以管理成员和分配任务。
3. 正式队员可以发布任务。
4. 每个人只能更新自己参与的任务。
5. 高难度技术任务需要所属方向组长或管理员批准后才能加入。
6. 任务在任何时刻都可以退出，但退出后默认不获得点数。
7. 退出任务记录和逾期任务记录只对本人和管理层可见，避免公开羞辱。
8. 待审核用户不可进入正式系统，只能查看审核状态和完善个人资料。

### 7.2 成员状态

| 状态 | 说明 |
|---|---|
| 待审核 | 注册后等待管理员审核 |
| 正常 | 可正常领取、参与、发布符合权限的任务 |
| 已退休 | 保留历史数据，只读浏览，不再参与任务与积分 |
| 已停用 | 账号被管理员停用，不可进入工作台 |
| 已退出 | 保留历史数据，但不再参与任务 |

### 7.3 成员管理功能

#### 管理员功能

1. 查看成员列表。
2. 搜索成员。
3. 按身份、部门、方向、状态、技能筛选成员。
4. 新增成员。
5. 编辑成员资料。
6. 审核注册申请。
7. 分配成员身份和权限角色。
8. 设置成员部门、方向、兵种组。
9. 禁用或恢复成员账号。
10. 查看成员积分明细。
11. 查看成员退出任务和逾期任务记录。
12. 记录成员晋升。

#### 普通成员功能

1. 查看成员公开资料。
2. 查看他人当前任务和公开贡献数据。
3. 编辑自己的头像、手机号、技能标签、个人简介。
4. 查看自己的积分历史、退出任务记录、逾期任务记录和晋升记录。

### 8.2 注册流程

```text
用户注册
→ 填写基础信息
→ 系统创建待审核账号
→ 管理员审核
→ 管理员分配身份、部门、权限
→ 用户进入系统
```

### 8.3 登录需求

1. 支持邮箱 + 密码登录（SHA256 加密存储）。
2. 支持用户名展示。
3. 登录页提供跳转注册入口。
4. 未审核用户登录后进入"审核中"页面。
5. 已停用用户登录后进入"已停用"提示页。
6. 密码错误时给出明确提示。
7. V1 可暂不实现验证码、第三方登录和忘记密码。

### 9.2 任务状态

| 状态 | 说明 |
|---|---|
| 待开始 | 任务已发布，但尚未开始 |
| 进行中 | 任务已有人参与并正在推进 |
| 待审核 | 任务成果已提交，等待管理员审核 |
| 已完成 | 管理员审核通过并完成点数结算 |
| 已逾期 | 超过截止日期且未完成 |

### 9.3 状态规则

1. 任务创建后默认为"待开始"。
2. 成员领取或负责人启动任务后变为"进行中"。
3. 成员提交成果后变为"待审核"。
4. 管理员审核通过后变为"已完成"，并发放点数。
5. 超过截止日期未完成时，系统可自动标记为"已逾期"。
6. 逾期完成不扣点，但逾期记录会保留，仅本人和管理层可见。

| 类型 | 示例 |
|---|---|

| 运维维护任务 | 场地建设、机器人维修、物资整理 |
| 运营宣传任务 | 推文、剪辑、招新、参观接待 |
| 招商财务任务 | 赞助沟通、材料采购、经费记录 |
| 比赛筹备任务 | 赛前检查、物资清单、战术训练、操作手训练 |
| 紧急任务 | 赛前突发维修、临时物资准备、紧急测试 |
| 长期项目任务 | 兵种长期研发、训练体系搭建、资料库建设 |
| 物资采购 | 元器件、加工件、工具、耗材采购 |

| 值班任务 | 场地值班、训练值守、设备巡检 |
| 复盘总结 | 赛后复盘、训练复盘、项目阶段总结 |

### 9.5 任务难度等级

| 等级 | 说明 | 是否需要审批 |
|---|---|---:|---:|
| 入门 | 育苗队员可独立完成 | 否 |
| 普通 | 梯队队员可完成 | 否 |
| 进阶 | 建议正式队员带领 | 视配置 |
| 高难 | 涉及关键技术或较高风险 | 是 |
| 核心 | 只允许指定成员参与 | 是 |

## 10. 任务市场

### 10.1 定位

任务市场是完全公开的任务广场，所有正式进入系统的成员都可以查看全部公开任务，并基于自己的兴趣、技能、组别和当前负载主动参与。

### 10.2 任务市场分区

| 分区 | 说明 |
|---|---|
| 可领取任务 | 成员可直接加入的任务 |
| 招募中任务 | 需要多人协作，展示缺口人数和推荐人群 |
| 学习任务 | 适合育苗和梯队成员积累研习点 |
| 高难审批任务 | 需要所属方向组长或管理员批准后加入 |
| 紧急任务 | 高优先级、临近截止或突发任务 |
| 长期项目 | 持续周期较长，通常关联兵种项目或基础建设 |

### 10.3 市场任务卡片信息

1. 任务标题。
2. 任务类型。
3. 所属方向 / 部门 / 兵种。
4. 难度等级。
5. 推荐适合人群。
6. 截止日期。
7. 当前状态。
8. 当前参与人数 / 人数上限。
9. 任务收益：研习点、工时点、管理点。
10. 是否需要审批。
11. 发布者。
12. 快捷操作：领取 / 申请加入 / 查看详情。

### 10.4 领取规则

| 情况 | 规则 |
|---|---|
| 普通任务 | 成员可直接领取 |
| 高难技术任务 | 需要所属方向组长或管理员批准 |
| 核心任务 | 仅指定成员可加入，或经管理员批准 |
| 人数已满 | 不允许直接领取，可进入候补列表 |
| 育苗队员领取任务 | 可以领取所有任务，但高难任务需审批 |

### 10.5 退出规则

1. 任务在任何时刻都可以退出。
2. 退出成员默认不获得点数。
3. 若成员已有明确贡献，管理员可在审核时手动补偿点数。
4. 退出记录仅本人和管理层可见。
5. 退出后任务卡片释放对应参与名额。

### 10.6 中途加入规则

1. 中途加入成员正常参与任务流程。
2. 中途加入成员完成后获得自己应得份额的一半。
3. 该规则在审核结算时自动计算，管理员可手动调整。

## 11. 任务发布与审核流程

### 11.1 任务发布权限

1. 所有正式队员都可以发布任务。
2. 管理员、组长也可以发布任务。
3. 梯队队员和育苗队员默认不可发布任务，可作为后续扩展权限。

### 11.2 发布流程

```text
正式队员创建任务
→ 填写任务信息、点数、推荐人群、截止日期
→ 选择是否公开到任务市场
→ 系统发布任务
→ 成员领取或申请加入
→ 任务推进
→ 成员提交成果
→ 管理员审核
→ 管理员确认点数比例
→ 系统结算积分
→ 任务完成归档
```

### 11.3 高难任务审批流程

```text
成员申请加入高难任务
→ 系统识别所属方向
→ 通知所属方向组长和管理员
→ 组长或管理员审批
→ 审批通过后加入任务
→ 审批拒绝后保留申请记录
```

### 11.4 任务完成审核流程

```text
成员提交成果
→ 任务进入待审核
→ 管理员查看成果、附件、评论、参与记录
→ 管理员确认是否通过
→ 管理员确认点数分配比例
→ 管理员可添加补偿点数
→ 系统发放研习点、工时点、管理点
→ 任务状态变为已完成
```

### 11.5 审核结果

| 审核结果 | 说明 |
|---|---|
| 通过 | 发放点数，任务完成 |
| 驳回修改 | 回到进行中，成员补充成果 |
| 关闭任务 | 特殊情况终止，不发放或仅发放补偿点 |
| 部分通过 | 可对部分成员发放点数，对部分成员不发放 |

## 12. 积分体系

### 12.1 积分名称

| 积分 | 含义 |
|---|---|
| 研习点 | 表示技术学习、知识掌握、文档产出、培训答辩、研发成长 |
| 工时点 | 表示执行劳动、事务支持、场地建设、维护值班、运营执行 |
| 管理点 | 表示任务负责人在组织协调、推进、验收中的额外管理贡献 |
| 综合贡献 | 研习点 + 工时点 + 管理点 |

### 12.2 点数规则

1. 每个任务可以同时拥有研习点和工时点。
2. 任务负责人可额外获得管理点。
3. 任务总点数按成员分配。
4. 默认平均分配。
5. 管理员审核时可以手动调整分配比例。
6. 待审核通过后才发放点数。
7. 逾期完成不扣点。
8. 退出任务默认不获得点数。
9. 中途加入成员获得自己应得份额的一半。
10. 管理员允许手动补偿点数。

### 12.3 综合贡献算法

```text
综合贡献 = 研习点 + 工时点 + 管理点
```

### 12.4 默认平均分配示例

任务总收益：

- 研习点：20
- 工时点：10
- 管理点：4

参与成员：4 人，其中 1 人为负责人。

默认分配：

| 成员 | 研习点 | 工时点 | 管理点 |
|---|---|---:|---:|---:|
| 负责人 | 5 | 2.5 | 4 |
| 协作者 A | 5 | 2.5 | 0 |
| 协作者 B | 5 | 2.5 | 0 |
| 协作者 C | 5 | 2.5 | 0 |

### 12.5 中途加入示例

任务原本 4 人，每人应得：

- 研习点：5
- 工时点：2.5

若某成员中途加入，则获得：

- 研习点：2.5
- 工时点：1.25

管理员可在审核时根据实际贡献调整。

### 12.6 手动调整比例

管理员审核时可设置贡献比例，例如：

| 成员 | 贡献比例 | 说明 |
|---|---:|---|
| 负责人 | 40% | 主要推进 |
| 协作者 A | 30% | 核心执行 |
| 协作者 B | 20% | 辅助执行 |
| 协作者 C | 10% | 少量参与 |

系统根据比例分配研习点和工时点。管理点仍由负责人额外获得，除非管理员手动调整。

### 12.7 补偿点数

补偿点用于处理特殊情况：

1. 成员退出任务但已产生有效贡献。
2. 任务被关闭但成员已有投入。
3. 任务成果未完全达标，但部分工作有价值。
4. 任务点数设置明显偏低，需要补偿。

补偿点必须记录：

- 操作管理员。
- 补偿对象。
- 补偿类型：研习点 / 工时点 / 管理点。
- 补偿数值。
- 补偿原因。
- 操作时间。

## 15. 成员负载计算

### 15.1 负载指标

| 指标 | 说明 |
|---|---|
| 进行中任务数 | 当前参与且未完成的任务数量 |
| 临近截止任务数 | 距截止日期较近的任务数量 |
| 待审核任务数 | 已提交但未审核的任务数量 |
| 逾期任务数 | 超过截止日期未完成的任务数量 |
| 当前任务点数总量 | 当前参与任务对应的总潜在点数 |

### 15.2 负载等级

| 等级 | 判断参考 |
|---|---|
| 空闲 | 0-1 个进行中任务 |
| 正常 | 2-3 个进行中任务 |
| 偏忙 | 4-5 个进行中任务，或存在多个临近截止任务 |
| 过载 | 6 个以上进行中任务，或存在多个逾期任务 |

### 15.3 使用场景

1. 管理员分配任务时查看成员负载。
2. 仪表盘展示团队整体负载分布。

## 21. 关键交互流程

### 21.1 成员注册审核流程

```text
用户注册
→ 填写资料
→ 进入待审核状态
→ 管理员在审核中心查看申请
→ 管理员设置成员身份、部门、权限
→ 审核通过
→ 用户获得系统访问权限
```

### 21.2 普通任务领取流程

```text
成员进入任务市场
→ 筛选任务
→ 查看任务详情
→ 点击领取
→ 系统检查人数上限和成员状态
→ 领取成功
→ 任务加入"我的任务"
```

### 21.3 高难任务加入流程

```text
成员进入任务市场
→ 查看高难任务
→ 点击申请加入
→ 填写申请说明
→ 所属方向组长或管理员审批
→ 审批通过后加入任务
```

### 21.4 任务退出流程

```text
成员进入任务详情
→ 点击退出任务
→ 系统提示退出后默认不获得点数
→ 成员确认退出
→ 系统记录退出日志
→ 释放任务名额
```

### 21.5 任务结算流程

```text
成员提交成果
→ 任务进入待审核
→ 管理员审核成果
→ 管理员确认贡献比例
→ 系统计算研习点、工时点、管理点
→ 管理员确认结算
→ 系统生成积分流水
→ 任务变为已完成
```

### 21.6 成员退休流程

```text
成员发起退休申请
→ 系统检查是否有未完成的任务
→ 有：提示先完成或移交任务
→ 无：确认退休
→ 成员标记为已退休
→ 历史数据保留，只读浏览
```

### 21.7 管理员停用/恢复流程

```text
管理员打开成员资料
→ 点击停用账号
→ 输入管理员密码确认
→ 成员标记为已停用
→ 无法登录系统

管理员打开成员资料
→ 点击恢复账号
→ 输入管理员密码确认
→ 成员恢复正常
```

## 22. 边界情况与规则

### 22.1 人数上限

1. 任务设置人数上限后，超过上限不可直接加入。
2. 可进入候补列表，后续版本实现。
3. 管理员可以强制添加成员。

### 22.2 点数小数

1. 平均分配可能产生小数。
2 系统可保留 1 位小数。
3. 排行榜按实际数值排序。

### 22.3 任务关闭

1. 管理员可关闭任务。
2. 关闭任务时必须填写原因。
3. 可选择不发点或发放补偿点。

### 22.4 成员身份变化

1. 成员晋升后保留历史任务和积分。
2. 晋升记录进入个人中心。
3. 身份变化不影响已获得点数。
4. 管理员可审核变岗申请（目标身份变更）。

### 22.5 成员退出战队

1. 成员状态变更为已退出。
2. 历史任务和积分保留。
3. 不再参与新任务。
4. 排行榜可选择是否包含已退出成员。

## 25. 技术架构

### 25.1 技术栈

| 层级 | 技术 | 说明 |
|---|---|---|
| 前端 | 原生 ES Modules | 无框架，纯 Vanilla JS |
| 后端 | Python FastAPI + Uvicorn | REST API |
| 数据 | SQLite（单表 JSON 快照） | 本地文件存储 |
| 样式 | 原生 CSS | 4 文件分模块（base / layout / components / responsive） |

### 25.2 架构特点

1. **共享快照模式**：整个应用状态以 JSON 形式存储在 SQLite 单表中，前端拉取/推送完整快照。
2. **乐观更新 + 版本冲突检测**：前端本地修改后立即渲染，保存时带版本号，冲突时提示用户。
3. **自动备份**：每 5 分钟生成时间戳备份，保留最近 24 个 + 1 个滚动最新备份。
4. **版本检测**：前端轮询 index.html 的 BUILD_VERSION，发现新版本时提示刷新。
5. **隐藏管理员**：首次启动自动生成随机凭据，不在成员目录中暴露。

### 25.3 目录结构

```

├── frontend/               # 前端模板入口（index, app, sw, bundle）
├── server.py               # FastAPI 后端服务
```

## 28. 总结

醒狮战队协作中枢的核心价值在于：

1. 用公开任务市场提升成员主动参与度。
2. 用研习点和工时点同时衡量学习成长与实际劳动。
3. 用管理点认可任务负责人的组织贡献。
4. 用审批和结算机制保证贡献记录相对公平。
5. 用成员负载和兵种进度帮助管理层掌握战队真实状态。
6. 用成员档案和晋升记录沉淀战队人才培养路径。

该系统应优先保证流程清晰、权限明确、点数规则可信、页面信息密度适中，并在视觉上保持黑夜毛玻璃的冷静克制风格。
